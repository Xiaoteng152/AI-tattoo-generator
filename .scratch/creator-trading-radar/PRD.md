# PRD：交易博主雷达

Status: ready-for-agent

## Problem Statement

用户会固定关注一组在 X/Twitter 发布交易观点的博主。目前需要逐个打开博主主页，判断哪些推文是新内容，再自行提取标的、方向、入场条件和风险。这一过程重复、容易遗漏，也很难保留“之前看过什么”的状态。

用户需要一个只针对交易场景的个人 Web 工作台：自定义关注博主，持续收集他们的新推文，记录未读与已读状态，并把当前未读推文压缩成精简、可追溯、不过度推断的交易信号。系统的职责是整理原文和对照用户自己的交易规则，不是替用户编造入场计划或自动执行交易。

## Solution

新增独立的“交易博主雷达”工作流与三栏 Web 页面：

- 左栏管理和筛选博主，并以折叠列表展示新增推文的一行预览。
- 中栏展示所选博主的推文，默认只看新增，可切换查看全部历史，并记录点击已读状态。
- 右栏自动总结所选范围内最新的未读推文，最多输出 3 条短摘要和 3 个有原文依据的交易信号。
- 每 15 分钟通过官方 X 用户时间线做一次增量同步，并支持手动刷新。
- 明确交易信号通过 Telegram 推送；符合策略和与策略冲突的信号都推送并明确标记。
- 现有 Supabase Postgres 继续作为唯一线上业务数据库；Cloudflare Worker 只负责每 15 分钟触发同步，不保存交易雷达业务数据。

该功能复用现有 Growth Automation Harness 的 Extraction、Raw Item、Normalization、Enrichment、Workflow Run 和 Execution Channel 思路，但建立独立的交易分析边界，不继续使用 AI tattoo generator 的增长分析 Prompt。

## User Stories

1. 作为个人交易者，我想在一个 Web 页面里查看关注博主的新推文，以便不再逐个打开 X 主页。

3. 作为个人交易者，我想粘贴 X 主页链接添加博主，以便不必手动截取用户名。
4. 作为个人交易者，我想在添加前校验账号是否存在，以便避免创建无效订阅。
5. 作为个人交易者，我想停用或恢复不再关注的博主，以便控制信息范围，同时保留已有推文和分析证据。

## Implementation Decisions

### 产品与页面

- 新增独立的交易博主雷达页面，不替换现有 Dashboard、DeepSearch 或增长工作流。
- 页面是桌面优先的三栏布局：左栏博主与新增导航，中栏推文流，右栏 AI 摘要和信号卡。
- 左栏支持多选；首次进入默认选择最近有更新的博主。用户可以继续勾选其他博主扩大当前范围。
- 左栏新增折叠区只显示发布时间和单行预览。点击后定位并高亮中栏项目，不复制完整正文或 AI 结论。
- 中栏默认过滤为“新增”，可以切换“全部”。已读项目保留并弱化显示。
- 单击推文或主动“全部标为已读”才修改阅读状态；仅选择或打开博主不自动清空新增数量。
- 右栏总结当前选择范围内按发布时间倒序的未读内容，总输入上限为最新 10 条。
- 右栏摘要最多 3 条，每条最多 40 个中文字；交易信号最多 3 张。
- 右栏标题区提供“策略设置”和“重新分析”。策略设置使用弹窗，不长期占用三栏空间。
- 窄屏按左、中、右顺序纵向排列，只保证功能可用，不为第一版设计独立移动端交互。

### 指定博主增量同步模块

- 建立独立的 Creator Timeline Connector。它接收平台用户 ID、增量游标、数量上限和内容过滤选项，返回原始帖子、抓取元数据和下一游标。
- 添加博主时接受 `@handle` 或标准 X 主页 URL，统一规范化为小写 handle，并通过官方 X 用户查询解析稳定的平台用户 ID。
- 首次同步从官方用户时间线导入最近 10 条符合条件的帖子；这些记录标记为首次导入，不计入后续“新增”数量。
- 后续同步使用已保存的最新 Post ID 作为 `since_id`，避免重复读取历史内容。
- 收录原创推文和引用推文，排除普通转推和回复。
- 每 15 分钟由受保护的定时入口触发同步，并提供用户主动触发的立即刷新入口。
- 指定博主同步不使用 SoPilot 热帖作为兜底。SoPilot 无法保证覆盖自定义账号，降级使用会制造“没有更新”的错误结论。
- X API 返回鉴权、余额、限流或网络错误时保留上次成功游标，记录结构化错误，不推进同步状态。
- 单个博主失败不阻断其他博主同步。页面展示每个博主的上次成功时间和最近错误。

### Raw Item 与监控状态

- 原始 X payload 必须在转换前保存，继续遵守 Raw Item 优先的证据链原则。
- 原始帖子需要持久化发布时间、作者平台 ID、帖子类型和首次发现时间，供时间排序、去重和信号追溯使用。
- 现有 Raw Item 同时承担“全局唯一帖子”和“只属于一个 Workflow Run”两个冲突职责。为避免扩大本功能范围或改坏旧工作流，交易雷达使用独立的 Creator Raw Item 存储原始 X payload，并由它承担持续监控、去重和阅读状态。
- 关注博主与 Creator Raw Item 之间建立稳定关联，并保存是否首次导入、发现时间和阅读时间。
- 阅读状态与抓取状态分离。同步游标不依赖用户是否阅读，标记已读也不会删除原始数据。
- X 互动指标可以保留在原始 payload 中；第一版不要求再维护一份独立、重复的 metrics 数据。
- 停用博主只停止后续同步，不删除该博主、历史推文或 Digest 来源。第一版主界面不提供物理删除。
- 第一版是单用户，但顶层记录保留未来增加 owner 归属的迁移空间；不在本 PRD 实现团队或租户模型。

### 持久化边界

- Supabase Postgres 是交易博主雷达唯一的线上业务数据库，继续复用现有 Prisma Persistence layer。
- 第一版只持久化五类业务状态：关注博主与同步游标、Creator Raw Item 与阅读状态、当前交易策略、AI Digest 与来源、Telegram 投递幂等记录。
- 入场价、入场时间、失效条件和策略匹配属于 Digest 中的结构化信号，不建立实时行情、K 线、仓位、订单或盈亏数据表。
- 当前选中的博主、折叠状态、“新增/全部”筛选和高亮项属于浏览器临时状态，不写数据库；未读数量由阅读时间实时聚合。
- X、OpenAI、Telegram 和 Cron 凭证只保存在部署平台的环境变量或 Secret 中，不写数据库。
- 第一版不使用 Cloudflare D1、Hyperdrive、KV 或 R2，也不做 PostgreSQL 与 Cloudflare 的双写。
- 第一版不自动清理 Raw Item、Digest 或投递记录。数据规模达到需要治理的程度后，再单独设计保留期和归档，不提前引入 R2。

### 交易信号分析模块

- 建立独立的 Trading Signal Analyzer，不复用硬编码的 AI tattoo 增长 Prompt。
- 模块输入是最多 10 条已归一化推文、当前交易规则文本、Prompt 版本和分析时间；输出必须经过结构化 schema 校验。
- AI 接口继续采用 OpenAI-compatible 配置：API Key、Base URL 和 Model 均由环境配置提供。
- 未配置 AI Key 时返回明确的未配置状态，不调用规则回退生成交易信号。
- 每次同步发现新增内容后，按当前未读范围自动生成一次分析并缓存。切换筛选不重复调用；新增内容或手动重新分析才创建新结果。
- 标记已读不会触发重新分析，避免阅读动作造成模型调用风暴。
- 摘要只陈述这批推文的核心观点，最多 3 条、每条最多 40 个中文字。
- 每个交易信号固定包含：标的、方向、入场价、入场时间、失效条件、策略匹配、匹配理由和来源推文。
- 方向只允许明确的 `LONG`、`SHORT`、`WATCH` 或 `NONE`。模型不能把一般看法强行转换为多空信号。
- 入场价、入场时间和失效条件必须有来源推文支持。原文没有明确表达时固定输出“未明确”。
- 入场时间允许保留条件式原文，例如“4H K 线收盘确认后”“回踩 68,000 时”。只有原文给出绝对时间时才做明确时间归一化。
- 信号按原文明确程度排序。不得按模型推测的潜在收益率、胜率或紧迫度排序。
- 来源必须指向实际参与该信号判断的推文；信号卡可以回到原推核验。
- 没有明确交易信号时允许输出空信号列表，不得为了填满 UI 生成内容。

### 交易规则配置

- 第一版只保存一份当前交易规则文本和版本号，不实现策略 DSL、回测或自动优化。
- AI 只能根据用户提供的规则输出 `MATCH`、`CONFLICT` 或 `UNKNOWN`，并提供一句简短理由。
- AI 不得修改、补全或替用户生成交易规则。
- 保存新规则后只影响后续分析。每个 Digest 除策略版本外还必须保存当时的策略文本快照，使历史结果可以独立追溯。
- 用户可以显式重新分析，让同一批历史推文基于当前策略创建新的 Digest；不得覆盖旧 Digest。

### Telegram Execution Channel

- Telegram 是第一版唯一站外 Execution Channel。
- 只有分析结果包含明确的 `LONG`、`SHORT` 或 `WATCH` 信号时才考虑推送；没有明确方向的普通摘要不推送。
- `MATCH` 和 `CONFLICT` 都推送，分别使用清晰的符合与冲突标签。只有 `UNKNOWN` 且没有明确方向的内容不推送。
- 只有定时或手动同步发现的后续新增推文可以触发推送。首次导入和用户手动“重新分析”只更新页面，不发送历史信号。
- 单次推送最多包含 3 个信号、精简摘要和原推链接，不发送完整长文。
- 每个分析结果保存投递状态和稳定幂等键。定时重试、页面刷新和手动同步不得重复推送同一 Digest。
- Telegram 凭证未配置时保存“未配置”状态并在页面提示，不让投递失败反向破坏同步和 AI 分析。

### API 与权限

- 所有交易博主雷达页面数据和写操作均要求现有 Google 会话。
- Creator API 支持列表、添加、停用和恢复。第一版不暴露物理删除入口。
- Post API 支持按博主、是否未读和时间排序查询，并支持单条已读与当前范围全部已读。
- Sync API 支持手动同步单个或全部启用博主；Cron 入口必须校验独立密钥。
- Digest API 支持读取当前缓存结果和显式重新分析。
- Strategy API 支持读取和保存当前交易规则。
- 错误响应区分输入无效、账号不存在、X 鉴权/余额、限流、AI 未配置、Telegram 未配置和内部失败。

### 运行与部署

- 第一版使用低频轮询，不接 X Filtered Stream，也不要求常驻 worker。
- 定时同步必须是短批量、可重入、可部分成功的任务，由 Cloudflare Worker Cron 每 15 分钟调用受保护的 Next.js Sync API。
- Cloudflare Worker 只保存目标 API 地址和 Cron Secret；X、OpenAI、Telegram 凭证及全部业务数据仍留在 Vercel 与 Supabase 边界内。
- Vercel Pro 或 Enterprise 可以改用同频率 Vercel Cron，但同一环境只能启用一个定时来源，避免重复触发。Vercel Hobby 默认采用 Cloudflare Worker Cron。
- 应用运行时使用 Supabase pooled connection，数据库迁移使用 direct connection；Migration 不在 Vercel 请求运行时执行。
- 同步、分析和投递分别记录状态；后一步失败不会回滚已经安全保存的原始推文。
- 环境配置新增 Telegram Bot Token、Telegram Chat ID 和 Cron Secret；AI 沿用现有 OpenAI-compatible 变量。

## Testing Decisions

- 测试只断言外部行为和稳定接口，不锁定内部函数拆分、ORM 调用顺序或 Prompt 的完整文案。
- Creator handle 解析测试覆盖 `@handle`、X URL、大小写、尾部斜杠和无效输入。
- Creator Timeline Connector 使用伪造 HTTP 响应测试首次导入 10 条、`since_id` 增量、分页、原创/引用保留、回复/转推排除、游标只在成功后推进，以及单账号失败隔离。
- Raw Item 持久化测试覆盖跨多次同步去重、运行历史关联不丢失和发布时间保存。
- 博主停用测试覆盖停用后不再同步、恢复后继续沿用原游标、历史推文与 Digest 来源不被删除。
- 阅读状态测试覆盖“选择博主不标已读”“点击单条标已读”“全部标为已读”“已读仍可在全部列表查询”。
- Digest 输入选择测试覆盖多选博主、只选未读、全局最新 10 条和稳定时间排序。
- Trading Signal Analyzer 测试覆盖结构化输出校验、3 条摘要与 3 个信号上限、40 字限制、空信号允许、缺失字段固定为“未明确”和来源 URL 必须属于输入集合。
- 使用固定模型响应测试边界：原文没有入场价或时间时，分析层不得接受模型凭空补出的值。
- Strategy Match 测试覆盖 `MATCH`、`CONFLICT`、`UNKNOWN`，并验证策略版本和策略文本快照随 Digest 保存。
- Telegram 投递测试覆盖明确方向触发、符合与冲突均触发、无明确方向不触发、幂等键去重、未配置凭证不破坏主流程。
- API 测试覆盖未登录拒绝写操作、非法 handle、账号不存在和各外部服务错误映射。
- Cron 测试覆盖错误 Secret 被拒绝、正确 Secret 可触发、重复触发不产生重复 Raw Item 或 Telegram 投递。
- 页面交互验证覆盖三栏桌面布局、默认最近更新博主、多选、左栏定位中栏、新增/全部切换、已读弱化、策略弹窗和重新分析。
- 优先沿用仓库中 Connector 的 Node Test 测试风格和 DeepSearch 深模块测试模式；实现阶段执行类型检查、测试、Lint 和生产构建。

## Out of Scope

- 不接入 X 以外的数据源。
- 不做博主搜索、推荐、分组、排行榜或批量导入。
- 不收录普通转推和回复。
- 不做秒级实时流、常驻 X Stream 连接或 Enterprise Webhook。
- 不做原生桌面 App 或原生移动 App。
- 不做完整移动端专属交互。
- 不做多人协作、邀请、角色、团队或多租户。
- 不做自动交易、券商或交易所下单。
- 不根据外部行情验证入场价，不计算仓位、止盈、止损或预期收益。
- 不让 AI 自行生成博主未表达的入场计划。
- 不自动修改或优化用户的交易规则。
- 不做策略回测、交易日志和盈亏归因。
- 不做多渠道推送；第一版只有 Telegram。
- 不做每条普通推文的即时通知。
- 不做博主及其历史证据的物理删除，第一版只允许停用和恢复。
- 不引入 Cloudflare D1、Hyperdrive、KV、R2 或任何双数据库同步。
- 不在第一版实现自动 TTL、冷归档或历史数据清理任务。

## Further Notes

- 当前仓库已经具备官方 X Recent Search Connector、Raw Item 到 Enrichment 的基本链路、PostgreSQL/Prisma、Google 登录和工作流运行记录，因此主要新增工作是指定账号时间线、持续状态和交易专用分析。

- Vercel Hobby 的 15 分钟调度使用 Cloudflare Worker Cron；Cloudflare 不承担业务持久化，也不直接访问 Supabase。
- 所有信号必须保留原推来源，并在 UI 和 Telegram 中清楚提示这是基于博主公开内容的整理，不构成自动交易指令。
